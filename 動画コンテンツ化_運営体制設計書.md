# Lark講座 動画コンテンツ化 & 新運営体制 設計書

---

## この設計書の目的

現在のリアルタイム講義中心の運営を、**動画コンテンツ（自学自習）＋課題提出（採点フィードバック）＋ライブ（発表・メンタリング）**のハイブリッド型に移行する。

### 移行の背景

| 現状 | 課題 | 移行後 |
|------|------|--------|
| 講師がリアルタイムで全講義を実施 | 講師の稼働に依存。スケールしない | スキル教育は動画で。講師はメンタリングに集中 |
| 受講生は決まった日時に参加が必要 | スケジュールが合わず脱落する人がいる | 自分のペースで動画学習。課題で理解を確認 |
| 質問はチャットで随時対応 | 同じ質問が何度も来る | FAQは動画化。個別の質問だけチャットで対応 |
| 理解度の確認手段がない | 発表会まで進捗が見えない | 各回の課題提出で理解度を段階的に確認 |

### 新しい講座の構造

```
[動画コンテンツ]        [課題提出]           [ライブセッション]
 スキル・操作・設計の    → 動画で学んだ内容を  → 発表会・中間レビュー
 インプット               自分の手で実践          メンタリング
 （いつでも・何度でも）   （採点＋フィードバック）  （月1〜2回に集約）
```

## Part 1：動画コンテンツ一覧

### 分類の考え方

| 種別 | 内容 | 特徴 |
|------|------|------|
| **A：操作チュートリアル** | Larkの機能ごとの操作方法 | 画面収録。一度作れば長期間使える |
| **B：設計思考レクチャー** | なぜそう設計するか。思考プロセスの解説 | 講師の顔出し or スライド。講座の核心部分 |
| **C：ケーススタディ** | 実在の構築事例の解説 | 業種別・課題別。受講生の理解を深める |
| **D：ワーク解説** | 課題の取り組み方・提出のポイント | 課題開始前に視聴。何をどこまでやるかを明確に |

**L2-1-1｜レベル2の全体像：構築者への道**
種別：B（設計思考）｜10分

話す内容：
- レベル2のゴール：「自分で使える人」から「相手の組織に構築できる人」になること
- 3ヶ月のロードマップ：Month 1（機能習得）→ Month 2（設計力）→ Month 3（模擬案件で実践）
- 最終成果物：模擬案件のポートフォリオ（要件定義書・構築済み環境・納品資料・プレゼン）
- レベル1との違い：視座の転換。「自分の感覚で作る」から「相手の言葉にならない課題を引き出して設計する」へ
- 課題提出のルールと評価基準の説明
- レベル1卒業生と直接受講者が混在する前提。「基礎に不安がある人はL2-1-2を視聴、自信がある人はスキップ」

**L2-2-1｜承認フローとは？組織にしかない「決裁」の仕組み**
種別：B（設計思考）｜8分

話す内容：
- なぜレベル1で扱わなかったか：1人社長には「承認する相手」がいない
- 承認フローが必要な場面：経費申請（5万円以上は上長承認）、休暇申請、稟議
- 組織における承認の本質：「誰かが確認・判断してOKを出すプロセス」
- 紙やメールで承認していた時代の問題点 → Larkで解決する方法
- 承認フローの構成要素：申請者 → 承認者 → 通知 → 記録
- 「クライアントの会社にはこれが必要になる。だからレベル2で学ぶ」という視座

---

**L2-2-2｜承認フローの構築：経費申請を作ってみる**
種別：A（操作）｜12分

話す内容：
- Larkの承認フロー設定画面の開き方
- 実演：経費申請フローの構築
  - フォームの設計（申請者名・金額・用途・添付ファイル）
  - 承認者の設定（固定の承認者を指定）
  - 承認後の通知設定（申請者に結果を通知）
  - 却下時の差し戻し設定
- テスト実行：自分で申請 → 自分で承認 → 通知が届くことを確認
- 完成した承認フローの動作フローを図で整理

**L2-2-3｜条件分岐付き承認：金額帯で承認者を変える**
種別：A（操作）｜10分

話す内容：
- 実務では「金額によって承認者が変わる」ケースが多い
- 実演：条件分岐の設定
  - 5万円未満 → リーダーが承認
  - 5万円以上10万円未満 → 部長が承認
  - 10万円以上 → 社長が承認
- 設定手順を画面収録でステップバイステップ
- テスト実行：3つの金額パターンでそれぞれ申請し、正しい承認者に流れることを確認
- 条件分岐を設計するときのコツ：「現実の運用ルールを先にヒアリングしてから設定する」

**L2-2-4｜フロー機能：人間の判断を挟む自動化**
種別：A（操作）｜10分

話す内容：
- レベル1の自動化との違い：レベル1は「機械が全自動で処理」、フロー機能は「途中で人間が判断する」
- 実演：見積書の承認フロー
  - トリガー：Baseに新しい見積レコードが追加
  - 自動アクション：担当上長にチャット通知
  - 人間の判断：上長が「承認」or「差し戻し」を選択
  - 承認後の自動アクション：ステータスを「承認済み」に変更 + クライアントに通知
- 「自動化」と「承認フロー」の組み合わせパターンの整理

**L2-5-2｜ヒアリングの型：As-Is / To-Beの引き出し方**
種別：B（設計思考）｜12分

話す内容：
- ヒアリングの目的：相手の「現状（As-Is）」と「理想（To-Be）」のギャップを明確にすること

**L2-8-1｜なぜ一気に作らないのか：MVPの考え方**
種別：B（設計思考）｜8分

話す内容：
- MVPとは：Minimum Viable Product = 最小限で価値が出るもの
- なぜ段階的に作るのか

  - 小さく作ると→すぐ使える→フィードバックが来る→改善しながら拡張
- クライアントの心理：「まだ全部じゃないの？」→ 「Phase 1で成果が出たから、Phase 2も頼む」に変わる
- Phase分けの考え方
  - Phase 1（導入）：1つのメインBaseで成功体験を作る（2-4週間）
  - Phase 2（拡張）：関連するBaseを追加（1-2ヶ月）
  - Phase 3（最適化）：自動化・ダッシュボード・権限の高度化（1-2ヶ月）
- 「最初から完璧を目指さない」は受講生自身の構築にも当てはまる

## Part 2：動画の目的別マトリクス

### なぜ動画にするもの / しないものがあるのか

## Part 3：課題提出・採点・フィードバック体制

### 新しい受講フロー

```
Week の流れ（例：レベル2 第2回）

月曜     動画4本を視聴（承認フロー & フロー設計）
         ↓ 自分のペースで学習
水〜金   課題に取り組む（承認フロー構築）
         ↓
日曜     課題をLarkで提出
         ↓
翌火曜   講師が採点 & フィードバック返却
         ↓
         次の動画に進む
```

### 課題提出の仕組み

| 項目 | 内容 |
|------|------|
| **提出先** | Lark Base「課題管理テーブル」にレコード追加 |
| **提出物** | スクリーンショット / Lark環境の共有リンク / テンプレートファイル |
| **提出期限** | 各回の動画公開から7日以内（遅延は個別相談） |
| **採点者** | 担当講師（1講師あたり5〜7名を担当） |
| **返却期限** | 提出から48時間以内 |

### 採点基準（共通ルーブリック）

全課題に共通する5段階評価基準。

| 評価 | 基準 | アクション |
|------|------|-----------|
| **5（優秀）** | 要求を超えた品質。即実務で使えるレベル | コメントで称賛 + 次の課題へ |
| **4（合格）** | 要求を満たしている。軽微な改善点あり | 改善ポイントをコメント + 次の課題へ |
| **3（条件付き合格）** | 核心は理解しているが、不足がある | 具体的な改善指示を出し、再提出を促す |
| **2（再提出）** | 重要な要素が欠落または誤っている | 該当する動画を再視聴してから再提出 |
| **1（要面談）** | 根本的な理解不足が見られる | 個別面談を設定して対面でフォロー |

### 課題ごとの採点ポイント

#### レベル1の課題

| 課題 | 採点ポイント | 合格ライン |
|------|-----------|-----------|
| ① 業務課題マップ | 業務の棚卸しが具体的か / 課題とゴールが明確か | ゴール設定が具体的であること |
| ② Base構築 | 2テーブル以上あるか / ルックアップor双方向関連を使っているか / フィールド設計が適切か | テーブル間の関連が機能していること |
| ③ ビュー & ダッシュボード & 自動化 | ビュー3種以上 / ダッシュボードの見せ方 / 自動化が動作するか | 全要素が揃い動作すること |
| ④ 発表資料 | Before/Afterが明確か / 成果物の全体像が伝わるか | ライブ発表会に参加すること |

#### レベル2の課題

| 課題 | 採点ポイント | 合格ライン |
|------|-----------|-----------|
| ⑤ 環境整備 | Lark環境が整っているか / Baseの構造説明が他者に伝わるか | 他者が見て構造を理解できること |
| ⑥ 承認フロー | 承認プロセスが動作するか / 条件分岐が正しいか | 2つのフローが正しく動作すること |
| ⑦ ワークプレイス & 権限 | ポータルの情報設計 / 権限設計書の論理性 | 部門別の見せ分けが設計されていること |
| ⑧ 高度な自動化 | 条件分岐＋承認トリガーが組み合わさっているか | 一連のフローが正しく動作すること |
| ⑨ ヒアリングシート | 課題の構造化ができているか / As-Is/To-Beの整理 | 課題→優先順位の流れが論理的であること |
| ⑩ 要件定義書 & 項目定義書 | スコープが明確か / テーブル設計が要件に合っているか | クライアントに見せて合意が取れるレベル |
| ⑪ 業種別テンプレート | 業種の課題を反映しているか / 実運用に耐える設計か | 業種固有の課題が設計に反映されていること |
| ⑫ 導入ロードマップ | Phase分けが合理的か / 中間レビューで発表 | ライブ中間レビューに参加すること |
| ⑬ 模擬案件 設計書 | ヒアリング→要件→設計の一貫性 | 設計書通りに構築が可能なレベル |
| ⑭ 模擬案件 構築 | 設計通りに動作するか / 承認・自動化が機能するか | 全要素が動作すること |
| ⑮ 納品資料 & プレゼン | 運用マニュアルの品質 / プレゼンの構成 | ライブ修了発表会に参加すること |

### フィードバックの形式

| フィードバック方法 | 使い分け |
|----------------|---------|
| **テキストコメント**（Lark Base内） | 全課題に対して。評価 + 具体的な改善点 |
| **画面録画フィードバック**（3〜5分） | 評価2以下の再提出時。操作を見せながら説明 |
| **個別面談**（15〜30分） | 評価1（要面談）時。根本的な理解不足のフォロー |

### フィードバックのテンプレート

```
【課題⑥ 承認フロー & フロー設計】

■ 評価：4（合格）

■ 良かった点
・経費申請のフローが実務で使えるレベルに仕上がっている
・条件分岐（10万円以上は部長承認）の設定が正しい

■ 改善ポイント
・承認却下時の「差し戻し通知」が設定されていない
  → 承認フローの設定画面で「却下時のアクション」を追加してください
・フロー名が「テスト」のまま → 「経費申請承認フロー」等の命名を

■ 次の課題に進んでOKです。上記の改善は時間があるときに対応してください。
```
